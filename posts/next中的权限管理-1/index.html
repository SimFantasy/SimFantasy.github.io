Next中的权限管理(一)

Simz
2024-08-08
React Nextjs

Next.js 新版本带了很多新特性，从Next.js 13引入App Router就和12版本的Pages Router有了诸多区别，多了很多页面状态not-found、error、loading等，在14中引入了React中的Server Component，让一些原本需要写到api中的服务端请求，可以直接在服务端组件中直接请求了。NextJs15也马上要出来了，又会有一些新的特性加进来，也很让人期待。

权限管理的方式

Authentication英文意思为身份验证，也就是我们所说的权限管理了。在Nextj.s的项目中最常用到的就是权限模块了，作为一个全栈的框架，Next.js也有诸多实现方式。

  • 可以通过本身的cookie、session来实现
  • 通过第三方的托管的Clerk、Kinde、Auth0等
  • 搭建项目自己的权限管理的Next-auth
  • 第三方框架自带的权限模块如：pocketbase、supabase等

项目说明

本文将会通过通过六个Task项目的来实现这五种权限管理的方式，

  • 使用 iron-session + prisma来实现自身cookie、session的身份验证
  • 使用 clerk+ prisma第三方托管来实现权限管理
  • 使用next-auth v4 + mongoose和next-auth v5 + prisma两个版本来实现权限管理
  • 使用pocketbase来实现身份验证
  • 使用lusia来实现身份验证

对Next.js进行权限管理后，还需要配合middleware.ts中间件的授权规则来实现对页面的权限配置。

再来middleware.ts

 1import { NextRequest, NextResponse } from 'next/server'
 2import { DEFAULT_LOGIN_REDIRECT, apiAuthPrefix, authRoutes, publicRoutes } from '@/lib/auth-routes'
 3import { getSession } from '@/actions/auth'
 4
 5export async function middleware(req: NextRequest) {
 6	const { nextUrl } = req
 7	const session = await getSession()
 8	const isAuth = session.isAuth
 9
10	const isApiAuthRoute = nextUrl.pathname.startsWith(apiAuthPrefix)
11	const isPublicRoute = publicRoutes.includes(nextUrl.pathname)
12	const isAuthRoute = authRoutes.includes(nextUrl.pathname)
13
14	if (isApiAuthRoute) {
15		return null
16	}
17
18	if (isAuthRoute) {
19		if (isAuth) {
20			return NextResponse.redirect(new URL(DEFAULT_LOGIN_REDIRECT, nextUrl))
21		}
22		return null
23	}
24
25	if (!isAuth && !isPublicRoute) {
26		return NextResponse.redirect(new URL('/login', nextUrl))
27	}
28
29	return null
30}
31

33	matcher: ['/((?!api|_next/static|_next/image|favicon.ico).*)']

Session的刷新时页面闪烁问题的处理

鉴于session这种权限方式会带来刷新时的页面闪烁问题，可以做个provider在页面加载时获取session信息，并且添加到store中，这样也可以增加在客户端的session获取，具体做法如下：

先增加一个hooks：

 1// /src/hooks/use-auth.ts
 2import { SessionData } from '@/lib/types'

 4

 6	authUser: SessionData | null
 7	isAuth: boolean

 9	setIsAuth: (isAuth: boolean) => void
10}

12export const useAuth = create<AuthState>()(set => ({
13	authUser: null,
14	isAuth: false,
15	setAuthUser: (authUser: SessionData | null) => set(() => ({ authUser })),
16	setIsAuth: (isAuth: boolean) => set(() => ({ isAuth }))
17}))

10	authUser: SessionData | null
11	children: React.ReactNode
12}
13
14export const AuthProvider = ({ authUser, children }: AuthProviderProps) => {
15	const [loading, setLoading] = useState(true)
16	const setIsAuth = useAuth(state => state.setIsAuth)
17	const setAuthUser = useAuth(state => state.setAuthUser)
18
19	useEffect(() => {
20		if (authUser) {
21			setAuthUser(authUser)
22			setIsAuth(true)
23			setLoading(false)
24		} else {
25			setAuthUser(null)
26			setIsAuth(false)
27			setLoading(false)
28		}
29	}, [authUser, setAuthUser, setIsAuth])
30
31	if (loading)
32		return (
33			<div className='flex justify-center items-center gap-x-2 w-fit h-6'>
34				<Loader size={24} className='animate-spin text-neutral-500' />
35				<span className='text-smtext-neutral-700'>Loading...</span>
36			</div>
37		)
38
39	return <>{children}</>
40}

注意事项

  1. 因为iron-session没有提供单独的客户端方法来获取session，所以在项目中，使用了api的方式获取，比较麻烦。

     1// /src/app/api/session/route.ts
 2
 3import { NextRequest } from 'next/server'
 4import { ApiResponse } from '@/lib/response'
 5import { getSession } from '@/actions/auth'
 6
 7export async function GET(req: NextRequest) {
 8	const session = await getSession()
 9
10	if (!session) {
11		return ApiResponse.unauthorized()
12	}
13
14	return ApiResponse.ok({
15		id: session.id,
16		username: session.username,
17		email: session.email,
18		image: session.image,
19		isAuth: session.isAuth
20	})
21}

  2. 同样的原因，logout也需要使用api方式，iron-session不能直接移除已经存在的cookie信息，虽然已经加密了，但是不能移除也很奇怪。

     1// /src/app/api/auth/logout/route.ts
 2

 4import { cookies } from 'next/headers'
 5import { ApiResponse } from '@/lib/response'
 6
 7import { getSession } from '@/actions/auth'
 8import { COOKIE_KEY } from '@/lib/constants'
 9
10export async function POST(req: NextRequest) {
11	const session = await getSession()
12
13	try {
14		session.destroy()
15		cookies().delete(COOKIE_KEY)
16		return ApiResponse.noContent()
17	} catch (error) {
18		return ApiResponse.badRequest('Logout failed')
19	}
20}

Iron-session项目总结

优点：

  1. 总的来说iron-session使用比较简单，通过简单的配置，即可实现session的权限搭建。

缺点：

  1. 也许就是因为比较简单，所以感觉使用起来麻烦，没有单独的客户端方法，获取就都必须走服务端或api。上面的注意事项就是这方面的问题。
  2. session的获取会有网络请求，所以就会有延迟，所以在每次刷新页面时就会出现闪烁，这样就需要多做一些交互状态。
  3. 另外因为iron-session会对cookie进行加密，如果有其他需要使用到权限相关的cookie时，这里需要另外处理。

最后，一些简单的小项目可以使用iron-session来做权限的处理，整体使用的效果不是太好。因为使用的cookie也加密了，与其他框架pocketbase不太好搭配使用。
