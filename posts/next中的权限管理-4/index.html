Next中的权限管理(四)

Simz
2024-08-08
React Nextjs

这篇我们来说说auth.js，也就是next-auth.js的v5版本。这个版本应该是根据next.js 13出现了App Router做了调整的版本，然后将之前一些方法做了合并，放到了auth这个函数上。具体可以查看官方说明：Authentication methods

毕竟是beta版本，很多东西会出现莫名其妙的错误，所以使用时需谨慎。

项目四：next-auth@beta + prisma实现Task项目的权限管理

项目初始化：

和之前next-auth类似，仅需要将next-auth@latest替换成next-auth@beta

 1# 使用pnpm，初始化Next.js项目
 2pnpm create next-app@latest next-auth-beta-prisma-demo
 3

10pnpm add -D prisma @types/bcryptjs

Auth Config

next-auth的配置都是写到一起了，在auth中对这个进行了拆分，将providers部分，拆了出来。

 1// src/lib/auth/auth-config.ts
 2
 3import type { NextAuthConfig } from 'next-auth'
 4import Credentials from 'next-auth/providers/credentials'
 5import { loginSchema } from '@/lib/schemas'
 6import { db } from '@/lib/db'
 7import { comparePassword } from '@/lib/utils'
 8
 9export default {
10	providers: [
11		Credentials({
12			async authorize(credentials) {
13				const validatedFields = loginSchema.safeParse(credentials)
14				if (!validatedFields.success) return null
15
16				const { email, password } = validatedFields.data
17
18				const user = await db.user.findUnique({ where: { email } })
19				if (!user || !user.password) return null
20
21				const isMatch = comparePassword(password, user.password)
22				if (!isMatch) return null
23
24				return user
25			}
26		})
27	]
28} satisfies NextAuthConfig

这里的内容和之前next-auth的基本一致，只是单独拿了出来，后续添加Github、Google的provider只需要在这个文件进行修改即可。

 1// src/lib/auth/index.ts
 2
 3import NextAuth from 'next-auth'
 4import { PrismaAdapter } from '@auth/prisma-adapter'
 5
 6import { db } from '@/lib/db'
 7import authConfig from '@/lib/auth/auth-config'
 8
 9export const {
10	handlers: { GET, POST },
11	signIn,
12	signOut,
13	auth
14} = NextAuth({
15	pages: {
16		signIn: '/login'
17	},
18	adapter: PrismaAdapter(db),
19	session: { strategy: 'jwt' },
20	callbacks: {
21		async signIn({ user, account }: { user: any; account: any }) {
22			if (account?.provider !== 'credentials') return true
23
24			if (user && user.id) {
25				const existingUser = await db.user.findUnique({ where: { id: user.id } })
26				if (!existingUser) return false
27			}
28
29			return true
30		},
31		async jwt({ token }: { token: any }) {
32			if (!token.sub) return token
33
34			const existingUser = await db.user.findUnique({ where: { id: token.sub } })
35			if (!existingUser) return token
36
37			token.id = existingUser.id
38			token.name = existingUser.name
39			token.email = existingUser.email
40			token.image = existingUser.image
41
42			return token
43		},
44		async session({ session, token }: { session: any; token: any }) {
45			if (token && session.user) {
46				session.user.id = token.id
47				session.user.name = token.name
48				session.user.email = token.email
49				session.user.image = token.image
50			}
51
52			return session
53		}
54	},
55	...authConfig
56})

这里才是auth的本体，最大的变化就是从NextAuth这里导出了handlers:{GET, POST}, signIn, signOut, auth几个方法。

  • handlers: {GET, POST}：这里要放到src/app/api/auth/[...nextauth]/route.ts中，和之前next-auth的做法类似。
  • signIn、signOut：这两个登录、登出的方法，之前是从next-auth/react中导出的，现在直接从这里导出
  • auth：这个是将之前提到诸多方法进行合并后的一个函数了，auth()这个方法还是一个服务端函数。

可以看到，这次user的action中添加了loginAction，其实只是为了统一下交互，直接使用signIn()在登录页面使用也是可以的，因为这里也是使用到了之前auth-config.ts配置中的providers中的authorize()。

然后要说下这个signOut()之前直接使用好像没啥问题，但这次直接使用无法登出，直接放到action中包裹下，就又正常了。

Middleware

在auth中的middleware与next-auth中的有一些变化。

 1// src/middleware.ts
 2
 3import { NextRequest, NextResponse } from 'next/server'
 4import NextAuth, { Session } from 'next-auth'
 5
 6import authConfig from '@/lib/auth/auth-config'
 7import {
 8	publishRoutes,
 9	authRoutes,
10	apiAuthPrefix,
11	DEFAULT_LOGIN_REDIRECT
12} from '@/lib/auth/auth-routes'
13
14export const { auth } = NextAuth(authConfig)
15
16type NextAuthRequest = NextRequest & { auth: Session | null }
17
18export default auth((req: NextAuthRequest) => {
19	const { nextUrl } = req
20	const isLoggedIn = !!req.auth
21
22	const isApiRoute = nextUrl.pathname.startsWith(apiAuthPrefix)
23	const isPublicRoute = publishRoutes.includes(nextUrl.pathname)
24	const isAuthRoute = authRoutes.includes(nextUrl.pathname)
25
26	if (isApiRoute) return
27
28	if (isAuthRoute) {
29		if (isLoggedIn) {
30			return NextResponse.redirect(new URL(DEFAULT_LOGIN_REDIRECT, nextUrl))
31		}
32		return
33	}
34
35	if (!isLoggedIn && !isPublicRoute) {
36		return NextResponse.redirect(new URL('/login', nextUrl))
37	}
38
39	return
40})
41
42export const config = {
43	matcher: ['/((?!.*\\..*|_next).*)', '/', '/(api|trpc)(.*)']
44}

从上述代码中可以看到，这次是直接从NextAuth(authConfig)中导出的auth来作为middleware的。

需要注意auth()的返回值不能返回null，之前在next-auth中我们都可以直接返回null，这里会报错。

Auth在服务端和客户端的方法

可以查看官方给的合并说明：Authentication methods

在客户端进保留了useSeesion()。

服务端则将之前的getServerSession()、getToken()都合并到了auth()中。

总结

优点：

  • 在next-auth的基础上做了些优化，但仍保留了next-auth的灵活性
  • 配置上比next-auth还是有些许简化。做了拆分，不用写一大堆内容在一个文件中。
  • auth.js的文档比next-auth.js的文档还是好了很多，但仍然很乱。

缺点：

  • 稳定性缺失，每次使用auth.js总会有些莫名的报错。
  • 在next.js 15版本中会报错，无法使用

总结：

auth.js可以在一些next.js 14的项目中尝试下，不建议在正式项目中使用，主要还是不稳定。

因为这个横向对比的项目，仅做了本地用户登录注册登出、以及在中间件上的使用。并没有展示出next-auth其他的一些特性，所以后续会再写一篇关于next-auth的完整使用的文章。
